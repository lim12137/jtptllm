# Global Concurrency Gate=1 Runtime Test Design

## 背景

当前已经为“HTTP 层全局总并发门闸”形成了设计与实施计划，但还需要一个最小、直观的运行态验证场景，来确认“超过全局并发上限时不是直接失败，而是进入等待队列”。

本次验证不追求吞吐数据，也不评估生产档位；它只是一个行为验证：把全局 gate 临时调成 `1`，然后并发发出 `5` 个 `/v1/chat/completions` 请求，观察是否只有第一个请求先进入执行、后 `4` 个请求排队等待，而不是同时进入或直接报错。

## 已确认范围

- 仅用于运行态验证
- 不是长期修改默认值
- 测试口径：
  - 全局 gate = `1`
  - 并发发出 `5` 个 `/v1/chat/completions` 请求
  - 观察后 `4` 个是否排队
- 不要求修改正式默认配置长期保留

## 设计目标

本次验证要回答一个非常明确的问题：

- 当全局 gate 设置为 `1` 时，系统是否真的表现为“同一时刻只允许 1 个请求进入执行，其余请求等待”？

因此，测试的成功标准不是“所有请求最终都成功”，而是：

1. 至少能观测到明显串行化/排队行为
2. 不出现“第 2-5 个请求直接被拒绝”的情况
3. 请求完成后，后续排队请求能够继续推进

## 为什么选 gate=1 + 5 个 chat 请求

这是最小又最直观的场景：

- gate=1 时最容易观察排队
- `5` 个请求足够看到“1 个执行 + 4 个等待”的效果
- 选择 `/v1/chat/completions`，避免同时混入另一条入口，减少变量
- 先验证行为，再考虑更高并发档位

## 运行态验证而非长期配置修改

本次变更是“临时测试配置”，不是产品默认值调整。

因此设计上必须明确：

- 临时值只用于验证窗口
- 测试结束后应恢复默认值 `16`
- 或者采用能避免长期保留临时值的方式（例如测试分支 / 临时 patch / 临时常量改动后立即回切）

## 推荐验证方式

### 方案 A：直接把默认常量改成 1，测试后再改回 16

- 优点：实现简单
- 缺点：如果收尾不严谨，最容易把临时值长期留在主工作区

### 方案 B：引入临时可配置入口，仅测试时传 1

- 优点：更安全
- 缺点：改动略大，不符合“最小运行态验证”目的

### 方案 C：在独立测试提交/工作树中临时改成 1，验证后立即恢复

- 优点：风险更低，最适合运行态验证
- 缺点：操作步骤稍多

推荐方案：C。

## 观测方式

建议同时观测三类信号：

### 1. 请求完成时间

如果 gate=1 生效，则 `5` 个并发请求的完成时间应呈现明显梯度，而不是几乎同时完成。

### 2. 代理日志

如果已有 IO/时间日志，可从日志中看到请求进入和返回顺序；至少应能验证在同一窗口里不是 5 个请求同时推进到上游执行。

### 3. 请求结果

重点看：

- 是否没有直接 `429`
- 是否没有因为 gate 满而立即返回并发错误
- 是否后续请求在前一个完成后继续推进

## 风险

### 1. 与现有 per-key queue 混淆

如果 5 个请求都使用同一 session key，可能同时受到：

- 第一层全局 gate=1
- 第二层 per-key pool/queue

为减少歧义，建议测试时使用不同 session key，让观察重点落在全局 gate。

### 2. 上游响应不稳定干扰结论

即使 gate 工作正常，请求仍可能因上游 `502` 或空响应失败。

因此本测试更关注：

- 请求是否等待
- 是否串行推进

而不是只看业务成功率。

### 3. 临时配置遗留

这是本次最大的操作风险：

- gate=1 是极端测试值
- 若遗留在主工作区，会严重影响正常验证与开发

因此实施计划必须包含明确恢复步骤。

## 验证要点

- chat 请求统一受全局 gate 控制
- gate=1 时同时发出 5 个请求，后 4 个应等待
- 不应直接返回 `429`
- 前一个请求结束后，下一个请求能继续推进
- 测试结束后恢复默认值 `16`，或确保临时值不留在长期工作区

## 验收标准

- 有一份专门针对“gate=1 + 5 个 chat 请求”的运行态验证设计
- 明确它只是临时运行态验证，不是默认配置变更
- 明确如何观测排队行为
- 明确如何恢复默认值或避免长期遗留临时配置
