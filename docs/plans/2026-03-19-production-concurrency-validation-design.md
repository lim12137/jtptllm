# Production Concurrency Validation Design

## 背景

当前代理已经具备按 session key 分池、多会话并发、池满排队、force new / closeAfter 轮换等并发机制。但现有验证资产存在两个缺口：

1. 运行态验证与代码级验证分离，缺少一套面向“生产可用并发级别”的统一口径。
2. 现有并发测试既有同 session key 压测，也有多 session 代码级验证，但尚未形成“不同 session key 下的总并发能力”基线。

本次目标不是证明理论峰值，而是得出一个可用于生产的保守并发档位。

## 已确认方案

- 目标：验证不同 session key 下的总并发能力
- 验收口径：生产保守口径
  - 成功率 `>= 99%`
  - 且 `p95` 延迟受控
- 方法：组合方案
  - 第一阶段：阶梯压测，找到候选上限
  - 第二阶段：对候选档位做稳态压测
- 成功定义：必须是“有效返回”
  - HTTP 200 但 `empty_200` 不算成功
- 失败分类至少包括：
  - `PROCESS_CONCURRENCY_LOCK`
  - `502`
  - `empty_200`
  - 超时
  - 其他
- 报告统一落到：`docs/reports/concurrency/`

## 设计目标

输出一套可重复、可对比、可保守决策的验证方法，用于回答：

- 当前代理在“不同 session key 隔离”的前提下，总并发可以稳定到多少？
- 哪个并发档位能作为生产黄金值，而不是偶发跑过的极限值？
- 失败主要来自上游锁、上游 `502`、空响应，还是客户端/网络超时？

## 测试对象与边界

### 测试对象

- HTTP 接口：`/v1/chat/completions`
- 请求模式：非 stream、非工具调用、轻量 prompt
- 并发模型：每个请求使用不同 session key，避免同 key 竞争影响结论

### 不在本次范围内

- 同 session key 并发上限
- tool call / stream 模式并发表现
- 上游 context length 极限
- 默认模型或模型映射行为验证

## 核心口径

### 成功定义

一个请求只有在同时满足以下条件时才算成功：

1. HTTP 状态是 `200`
2. 响应结构合法
3. 能提取到有效文本返回
4. 返回文本非空白

### 失败分类

所有非成功请求都必须归类，至少包括：

- `PROCESS_CONCURRENCY_LOCK`
  - 响应文本或错误信息中出现该锁标记
- `502`
  - HTTP `502` 或代理统一映射的 upstream `502`
- `empty_200`
  - HTTP `200`，但无有效文本
- `timeout`
  - 请求超时
- `other`
  - 其他未归入上述类别的错误

### 生产保守口径

“生产可用并发级别”不取“最高通过一次”，而取满足以下条件的最大档位：

- 成功率 `>= 99%`
- `p95` 延迟在预先设定的可接受范围内
- 失败分布无明显异常尖峰
- 稳态压测结果与阶梯压测一致，不出现明显回落

## 方法设计

### 方案 A：只做阶梯压测

- 优点：执行快，成本低
- 缺点：容易把瞬时幸运结果误判为稳定能力，不能代表生产保守值

### 方案 B：只做单档长稳态压测

- 优点：更接近生产
- 缺点：没有候选区间，容易选错档位，效率低

### 方案 C：阶梯压测 + 稳态压测

- 优点：先粗筛上限，再细验稳定性，兼顾效率和可靠性
- 缺点：执行时间更长，但结论质量明显更高

推荐采用方案 C。

## 分阶段设计

### 阶段 1：阶梯压测

目的：快速定位候选并发上限区间。

建议档位：

- `1`
- `2`
- `4`
- `6`
- `8`
- `10`
- `12`
- `16`
- `20`

如在某一档附近出现明显拐点，可加密测试，例如：

- `10`
- `12`
- `14`
- `16`

阶段 1 关注：

- 成功率趋势
- `p95` 变化趋势
- 失败分类分布
- 是否已出现明显不稳定区间

输出：

- 候选上限档位
- 候选保守档位
- 需要进入稳态验证的 1-3 个档位

### 阶段 2：稳态压测

目的：验证候选档位在更长时间内是否仍满足生产保守口径。

建议策略：

- 选择阶段 1 的 1-3 个候选档位
- 每档做固定总请求量或固定持续时间的稳态压测
- 每个请求使用唯一 session key，确保验证的是“总并发能力”而不是同 key 锁竞争

阶段 2 关注：

- 成功率是否仍 `>= 99%`
- `p95` 是否稳定受控
- 失败是否集中于少数类型
- 是否出现长尾恶化

输出：

- 最终推荐生产黄金并发值
- 不建议超过的风险区间
- 失败主因排名

## 数据与产物设计

### 建议脚本产物

- 原始 JSON 结果：每次运行保存在 `docs/reports/concurrency/raw/`
- 汇总 Markdown 报告：保存在 `docs/reports/concurrency/`

### 建议报告内容

每份报告至少包含：

- 测试日期、目标地址、模型、请求模式
- 使用的 session key 策略
- 阶梯压测命令摘要
- 稳态压测命令摘要
- 每档结果表：
  - 并发数
  - 总请求数
  - 成功数
  - 成功率
  - `empty_200`
  - `PROCESS_CONCURRENCY_LOCK`
  - `502`
  - `timeout`
  - `other`
  - `avg_ms`
  - `p95_ms`
- 结论：
  - 最大可过档位
  - 推荐生产黄金档位
  - 风险说明

## 判定规则

### 候选并发上限

满足以下条件时，可视为候选上限：

- 阶梯压测中成功率仍接近目标
- `p95` 尚未出现明显失控
- 失败分类中未出现大规模锁错误或空返回

### 生产黄金值

满足以下条件的最大稳态档位：

- 成功率 `>= 99%`
- `p95` 延迟受控
- `empty_200` 不构成主要失败来源
- `PROCESS_CONCURRENCY_LOCK` 与 `502` 未显著增长

若多个档位都满足，则取更高者；若边界波动明显，则向下取一档作为保守值。

## 风险与局限

- 上游稳定性会显著影响结论，尤其是 `502` 与锁错误
- 同一时间窗口的结果可能受上游负载波动影响，需要保留时间戳和环境说明
- 仅验证不同 session key 下的总并发，不代表同 key 并发能力
- 仅验证非 stream、非 toolcall、轻量请求；不能直接外推出复杂请求模式

## 验收标准

- 存在一份完整设计对应的执行方案
- 存在可重复执行的压测脚本与结果汇总方式
- 报告能区分成功与 `empty_200`
- 报告能按至少 5 类失败原因统计
- 最终能产出一个“生产保守黄金并发值”结论，而不是只报极限峰值
