# Concurrency Memory Validation Design

## 背景

当前仓库已经有“不同 session key 下总并发能力”的压测脚本与报告模板，但还缺少一套面向运行中 Go debug 代理内存占用的联动验证方法。现有运行形态下，代理通常由 `go.exe` 启动并派生临时路径下的 `proxy.exe`。如果只盯一个进程名，容易误判内存来源；如果简单把两者相加，又会模糊真正承载服务请求的进程。

因此，本次测试的目标不是求一个“总内存”单值，而是明确：在 `8/16/20` 并发档位下，`proxy.exe` 作为主服务进程的内存表现如何，同时 `go.exe` 作为调试运行链路的一部分是否有明显联动变化。

## 已确认方案

- 主结论看 `proxy.exe`
- 同时记录 `go.exe`
- 不把两者简单相加为唯一结论
- 压测驱动复用：`scripts/bench_concurrency_multi_key.ps1`
- 并发档位固定为：`8`、`16`、`20`
- 方法：采样对比 + 请求结果联动
- 报告统一落到：`docs/reports/concurrency/`

## 设计目标

本次验证要回答两个问题：

1. 在 `8/16/20` 并发下，`proxy.exe` 的内存占用是否随并发显著上升，是否存在不可接受的峰值。
2. 在相同压测窗口内，请求结果是否仍满足成功率与延迟要求；也就是说，不能只看内存低，还要同时看 `success_rate` 与 `p95` 是否稳定。

因此，最终结论必须基于“两类信号”联合判断：

- 请求结果信号：成功率、`p95`、失败分类
- 内存信号：`proxy.exe` 主指标 + `go.exe` 辅助指标

## 测试对象与边界

### 测试对象

- 压测入口：`/v1/chat/completions`
- 并发模式：不同 session key 下的总并发能力
- 代理运行形态：Go debug 代理（`go.exe` + `proxy.exe`）
- 并发档位：`8`、`16`、`20`

### 不在本次范围内

- stream/toolcall 模式
- 同 session key 并发行为
- 上下文长度测试
- `proxy.exe` 与 `go.exe` 的合并总内存口径
- 正式发布版单二进制进程内存分析

## 为什么主结论看 `proxy.exe`

当前 Go debug 运行形态下：

- `go.exe` 更接近调试/启动链路的一部分
- `proxy.exe` 才是承载 HTTP 请求、会话池和业务逻辑的实际服务进程

因此：

- 生产可读结论应以 `proxy.exe` 为主
- `go.exe` 只作为辅助观测对象，用来识别调试模式是否引入额外波动
- 禁止把 `proxy.exe` 与 `go.exe` 简单相加后作为唯一结论，因为这样会掩盖真正服务进程的变化趋势

## 采样与联动方法

### 方案 A：只看压测结果，不采内存

- 优点：执行最简单
- 缺点：无法回答 `16/20` 并发是否带来不可接受内存峰值

### 方案 B：只看内存曲线，不关联请求结果

- 优点：可以看到峰值
- 缺点：无法判断高峰值是否同时伴随成功率崩坏；缺少生产决策意义

### 方案 C：请求结果 + 内存采样联动

- 优点：能同时判断吞吐、稳定性和资源代价，最接近生产评估
- 缺点：执行与汇总更复杂

推荐采用方案 C。

## 指标设计

### 请求结果指标

沿用现有并发压测口径：

- `success`
- `PROCESS_CONCURRENCY_LOCK`
- `502`
- `empty_200`
- `timeout`
- `other`
- `success_rate`
- `p95_ms`

### 内存指标

至少记录以下指标：

- `WorkingSet64`
  - 当前驻留物理内存
- `PrivateMemorySize64`
  - 进程私有内存
- `PeakWorkingSet64`
  - 运行期间峰值驻留物理内存

### 进程范围

- 主指标：`proxy.exe`
- 辅助指标：`go.exe`

### 采样点

建议每个并发档位至少有三个关键采样阶段：

- `baseline`：压测前
- `during`：压测中，持续采样或定时采样
- `after`：压测后

并在结果汇总中提取：

- baseline 值
- 最大值（峰值）
- 结束值
- 相对 baseline 的增量

## 执行流程设计

### 阶段 1：确认目标进程

先识别当前运行实例中的：

- `go.exe` PID、路径
- `proxy.exe` PID、路径

必须在报告中明确记录，避免采错其他同名进程。

### 阶段 2：按并发档位执行压测与采样

对每个档位 `8/16/20`：

1. 记录 baseline 内存
2. 启动该档位压测
3. 在压测过程中采样 `proxy.exe` 与 `go.exe` 内存
4. 压测结束后记录 after 内存
5. 将请求结果与内存采样关联为同一档位结论

### 阶段 3：汇总与判定

每个档位同时回答：

- 请求是否稳定（成功率、`p95`）
- `proxy.exe` 峰值是否明显上升
- `go.exe` 是否同步有异常放大
- 是否值得作为生产保守档位

## 判定原则

### 单档位通过条件

一个并发档位要被视为“可接受”，至少要同时满足：

- `success_rate` 达到目标
- `p95_ms` 受控
- `empty_200` 不计入成功
- `proxy.exe` 峰值没有超出预设可接受范围
- `go.exe` 虽可上升，但不能出现异常失控增长

### 生产保守判断

最终推荐值不取“最高跑过一次”的档位，而取：

- 请求结果稳定
- `proxy.exe` 内存峰值可解释、可接受
- `go.exe` 变化不构成额外重大风险

若 `20` 并发成功率合格但 `proxy.exe` 峰值明显陡增，则可保守下调为 `16` 或 `8`。

## 建议报告结构

报告建议至少包含：

- 测试日期、目标地址、模型、代理运行方式
- `go.exe` / `proxy.exe` PID 与路径
- 并发档位：`8`、`16`、`20`
- 请求结果表：
  - `success`
  - `PROCESS_CONCURRENCY_LOCK`
  - `502`
  - `empty_200`
  - `timeout`
  - `other`
  - `success_rate`
  - `p95_ms`
- 内存结果表：
  - `proxy.exe` baseline / peak / after 的 `WorkingSet64`
  - `proxy.exe` baseline / peak / after 的 `PrivateMemorySize64`
  - `proxy.exe` 的 `PeakWorkingSet64`
  - `go.exe` 同样字段
- 结论：
  - 哪个档位内存增长最明显
  - `16` 并发是否仍在可接受区间
  - 推荐生产保守并发档位

## 风险与局限

- Go debug 模式的 `go.exe` 与正式发布版单进程形态不同，因此结论不能直接等同于生产发布版二进制
- Windows 进程内存存在瞬时波动，单点采样不够，应采用连续采样或多点采样
- 若压测窗口太短，`PeakWorkingSet64` 可能不稳定；需要合理的持续时间或总请求量
- 如果存在多个 `go.exe`/`proxy.exe` 实例，必须通过 PID 和路径精确区分

## 验收标准

- 有一套明确的 `8/16/20` 并发内存联动验证方案
- 能区分 `proxy.exe` 与 `go.exe`，并以 `proxy.exe` 为主结论
- 内存结果与请求结果联动展示
- 报告落盘到 `docs/reports/concurrency/`
- 结论能同时评估成功率、`p95` 与内存峰值，而不是只看单一指标
