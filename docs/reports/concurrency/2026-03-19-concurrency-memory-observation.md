# 2026-03-19 Concurrency Memory Observation

## 范围与说明

- 本报告基于已有 raw 数据做阶段性观察，不改代码。
- 观测对象主结论仅基于 `proxy.exe`；`go.exe` 仅作为辅助参考。
- 使用的 raw 文件：
  - `docs/reports/concurrency/raw/2026-03-19_105434-concurrency-memory.json`
  - `docs/reports/concurrency/raw/2026-03-19_105749-concurrency-memory.json`
  - `docs/reports/concurrency/raw/2026-03-19_110105-concurrency-memory.json`
- 三档并发分别为 `8`、`16`、`20`。
- 三档 `benchmark_exit_code` 均为 `timeout`，因此本报告不对同轮 `success`、`p95`、可用稳态并发做结论。
- 这是一份阶段性内存观察，不是完整联动成功报告。

## 原始峰值摘录

### `proxy.exe` 峰值

| 并发 | WorkingSet64 | PrivateMemorySize64 | PeakWorkingSet64 |
| ---: | ---: | ---: | ---: |
| `8`  | `25,518,080` | `29,614,080` | `25,522,176` |
| `16` | `26,484,736` | `30,560,256` | `26,488,832` |
| `20` | `27,672,576` | `31,703,040` | `27,680,768` |

### `go.exe` 峰值（辅助）

| 并发 | WorkingSet64 | PrivateMemorySize64 | PeakWorkingSet64 |
| ---: | ---: | ---: | ---: |
| `8`  | `18,137,088` | `30,220,288` | `30,834,688` |
| `16` | `18,137,088` | `30,220,288` | `30,834,688` |
| `20` | `18,141,184` | `30,220,288` | `30,834,688` |

`go.exe` 三档变化极小，因此以下增幅分析只以 `proxy.exe` 为准。

## `proxy.exe` 增幅计算

### `8 -> 16`

- `WorkingSet64`: `25,518,080 -> 26,484,736`，增加 `966,656`，约 `+3.79%`
- `PrivateMemorySize64`: `29,614,080 -> 30,560,256`，增加 `946,176`，约 `+3.20%`
- `PeakWorkingSet64`: `25,522,176 -> 26,488,832`，增加 `966,656`，约 `+3.79%`

### `16 -> 20`

- `WorkingSet64`: `26,484,736 -> 27,672,576`，增加 `1,187,840`，约 `+4.48%`
- `PrivateMemorySize64`: `30,560,256 -> 31,703,040`，增加 `1,142,784`，约 `+3.74%`
- `PeakWorkingSet64`: `26,488,832 -> 27,680,768`，增加 `1,191,936`，约 `+4.50%`

## 观察

- 在这三份 raw 数据里，`proxy.exe` 的三项峰值指标都随并发档位从 `8 -> 16 -> 20` 单调上升。
- `8 -> 16` 的增幅已经是正向且一致的，不是单一指标偶发波动。
- `16 -> 20` 仍继续上升，且增幅略高于 `8 -> 16`。
- `go.exe` 作为辅助参考基本稳定，说明这组样本里主要增长信号仍集中在 `proxy.exe`。

## 能否回答“16 并发会增加内存”

可以，但结论范围必须收敛。

可以回答的是：

- 相比 `8` 并发，这组样本中的 `16` 并发确实对应更高的 `proxy.exe` 峰值内存。
- 这个“增加”至少同时体现在：
  - `WorkingSet64`
  - `PrivateMemorySize64`
  - `PeakWorkingSet64`

不能回答的是：

- `16` 并发在“联动测试成功完成”的前提下会增加多少内存。
- `16` 并发是否为可用生产稳态档位。
- 内存增长是否与成功吞吐、P95 延迟形成稳定对应关系。

原因：

- 三档 benchmark 都是 `timeout`
- 因此缺少同轮可靠的成功率、P95、有效返回数据，不能把内存观察升级为完整联动结论

## 阶段性结论

- 本阶段可以确认：在现有 raw 数据中，`proxy.exe` 的峰值内存随并发从 `8` 提升到 `16`、再到 `20` 而上升。
- 因而，对“`16` 并发会不会增加内存”这个问题，阶段性答案是：会。
- 但这仍然只是阶段性观察，不是完整联动成功报告；后续仍需要在非 `timeout` 条件下补齐同轮 benchmark 成功数据，才能把内存变化与真实并发表现关联起来。
